# Gate-3 decode — Lever 2: 64-bit entropy window on top of the entropy codegen chunk (2026-07-22)

Task #37, DECODE-PERF. Question posed by the coordinator: does the 64-bit
`od_ec` window + rav1d bulk-8-byte refill (preserved as the unlanded divergent
jj change `74d701a8`) compose ON TOP of the landed entropy hot-path codegen
(`a00aa516`, which kept the 32-bit window) for an *additional* bit-exact win?

**Answer: yes — it is a small but clean additional bit-exact win, and it makes
the port more faithful to C** (aom's own `od_ec.h` uses a `size_t`-width
`OD_EC_WINDOW`, i.e. 64-bit on the 64-bit build we benchmark against).

## What landed

`crates/aom-dsp/src/entropy/dec.rs`: window widened 32 → 64 bits (`dif: u32 →
u64`, `OD_EC_WINDOW_SIZE 32 → 64`), `refill` rewritten as a single masked
big-endian 8-byte load with the per-byte tail kept for the < 8-byte
end-of-buffer case (the dav1d/rav1d `ctx_refill` shape). The `#[cold]` /
never-inline `refill` wrapper and the check-free iterator `decode_cdf_q15`
scan from `a00aa516` are **preserved** — this composes the two, it does not
revert either. `normalize`/`decode_bool_q15`/`decode_cdf_q15` thread the wider
`dif` through with `as u32` / `as u64` at the decision points.

Decision-invariance (why it stays byte-exact): every symbol decision compares
`dif` against `vw = v << (W-16)` whose low `W-16` bits are zero, so
`dif >= vw ⟺ (dif >> (W-16)) >= v` for ANY window width `W`; bytes are consumed
in the same order; past the buffer end both widths consume all-ones filler and
latch on the same `cnt = OD_EC_LOTS_OF_BITS`.

## Measured (callgrind Ir, `gate3_profile dec port <cell> 3` = 1 C-oracle + 4 port decodes)

| cell | metric | baseline | w64 | Δ |
|---|---|---:|---:|---:|
| dec_352x288_q00 (lossless, entropy-dominated) | program total Ir | 510,619,494 | 506,476,248 | **-0.81%** |
| dec_352x288_q00 | port Ir/decode | 105,761,982 | 104,726,170 | **-0.98%** |
| dec_352x288_q32 (filter-dominated) | program total Ir | 238,921,186 | 238,445,542 | -0.20% |
| dec_352x288_q00 | `OdEcDec::refill` self-Ir | 10,575,564 | 2,135,492 | **-79.8%** |
| dec_352x288_q32 | `OdEcDec::refill` self-Ir | 1,229,976 | 248,660 | **-79.8%** |

The refill work drops ~80% (bulk load + ~half as many refills). The net
per-decode win is smaller than the refill saving because the wider `dif`
adds a little per-symbol arithmetic to the inlined `decode_cdf_q15` /
`decode_bool_q15` (measured +3.3 M in the q00 inlined decode blob), so the
net is ~1% on the entropy-heavy cell and ~0.2% on the filter-heavy cell.
The mosaic 2k/4k photographic cells (where entropy is a larger share and the
stream has more bytes → more refills) are gitignored and were not measurable
this session; the win there is expected to be between these two bounds but is
NOT reported as a number (no extrapolation).

## Bit-exactness proof

* `entropy_diff.rs` (the C-differential vs the REAL `od_ec_dec`): random op
  sequences + roundtrip + the truncated-buffer sequences (decode far past the
  end of the buffer — the regime where the 64-bit prefetch behaves most
  differently internally from C's 32-bit one). All 3 pass with the widened
  window.
* Full decode conformance `conformance_single_frame_intra_byte_identical_to_c_and_golden`
  (240 vectors, byte-identity + golden MD5) — green. Every decoded symbol goes
  through this range decoder, so a single wrong bit fails the whole corpus.
